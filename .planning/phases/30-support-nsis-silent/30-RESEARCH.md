# Phase 30: Support-NSIS-Silent - Research

**Researched:** 2026-02-28
**Domain:** NSIS installer silent mode, electron-builder, electron-updater
**Confidence:** HIGH

## Summary

The `SetSilent normal` line in `customInit` is the root cause of both broken `/S` silent installs and likely broken auto-updates. electron-updater's `NsisUpdater.doInstall()` passes `/S` when `isSilent=true`, and `autoInstallOnAppQuit` calls `install(true, false)` -- meaning every auto-update attempt runs the new installer with `/S`, which gets overridden back to `normal` by `customInit`, causing the full wizard UI to appear during what should be a seamless background update.

The fix is straightforward: remove `SetSilent normal`, and add silent-mode detection to write a marker file so the app can skip the setup wizard on first launch after a silent install.

**Primary recommendation:** Remove `SetSilent normal` from `customInit`. Use `${Silent}` NSIS built-in and `${isUpdated}` electron-builder flag to conditionally write a `silent-install` marker file to `$LOCALAPPDATA\claude-terminal\`. In the app, detect this marker in `isFirstLaunch()` and auto-apply defaults without showing the wizard.

## Architecture Patterns

### How electron-builder NSIS Flags Work (HIGH confidence)

electron-builder auto-generates NSIS macros for command-line flags via `nsisScriptGenerator.js`. The `flags()` method creates LogicLib-style conditionals:

```nsis
; Generated from flags(["updated", "force-run", ...])
; Each flag gets a macro using StdUtils.TestParameter

; "updated" -> ${isUpdated} - checks for --updated on command line
; "force-run" -> ${isForceRun} - checks for --force-run on command line
; "keep-shortcuts" -> ${isKeepShortcuts}
; "no-desktop-shortcut" -> ${isNoDesktopShortcut}
; "delete-app-data" -> ${isDeleteAppData}
; "allusers" -> ${isForAllUsers}
; "currentuser" -> ${isForCurrentUser}
```

The variable name is generated by: `"is" + flagName[0].toUpperCase() + camelCase(rest)`.
These use `StdUtils.TestParameter` internally to check if `--flagname` was passed on the installer command line.

**Source:** `node_modules/app-builder-lib/out/targets/nsis/nsisScriptGenerator.js` (local verification, HIGH confidence)

### How electron-updater Invokes the Installer (HIGH confidence)

`NsisUpdater.doInstall()` in `electron-updater/out/NsisUpdater.js` constructs the installer command:

```javascript
// Always present:
const args = ["--updated"];

// When isSilent=true:
args.push("/S");

// When isForceRunAfter=true:
args.push("--force-run");
```

Three call paths exist:

1. **`autoUpdater.quitAndInstall()`** (called by UpdaterService) -> `quitAndInstall(isSilent=false, isForceRunAfter=false)` -> installer runs with `--updated` only (wizard mode, no force-run)
2. **`autoInstallOnAppQuit`** handler (on app quit) -> `install(true, false)` -> installer runs with `--updated /S` (silent, no force-run)
3. **Explicit `quitAndInstall(true, true)`** -> installer runs with `--updated /S --force-run` (silent with app relaunch)

**Critical finding:** The current `SetSilent normal` in `customInit` overrides the `/S` flag in paths 2 and 3, breaking auto-update-on-quit (forces wizard UI to appear when user is trying to close the app).

**Source:** `node_modules/electron-updater/out/NsisUpdater.js:101-145`, `node_modules/electron-updater/out/BaseUpdater.js:12-89` (local verification, HIGH confidence)

From `installer.nsi`, the `.onInit` function runs:

### Post-Install App Launch Logic (HIGH confidence)

From `installSection.nsh` lines 91-110, for assisted installer (oneClick=false):

```nsis
; For assisted installer, app launch ONLY happens if BOTH conditions are true:
${if} ${isForceRun}
${andIf} ${Silent}
  !insertmacro doStartApp
${endIf}
```

This means:
- **Normal wizard install:** App launches via the MUI finish page "Run" checkbox (user choice)
- **Silent install without `--force-run`:** App does NOT launch (exactly what we want)
- **Silent install with `--force-run`:** App launches

For our use case (silent install = no auto-launch), this is already correct behavior. No changes needed to post-install launch logic.

**Source:** `node_modules/app-builder-lib/templates/nsis/installSection.nsh:104-109` (local verification, HIGH confidence)

### Assisted Installer Page Skip on Update (HIGH confidence)

The `skipPageIfUpdated` macro in `common.nsh` aborts (skips) pages when `${isUpdated}` is true. The license page and directory page use this. When installing silently (`/S`), NSIS automatically skips all pages anyway.

**Source:** `node_modules/app-builder-lib/templates/nsis/common.nsh:110-121` (local verification, HIGH confidence)

### Silent Install Signal to App (HIGH confidence)

**Recommended approach: Marker file in `$LOCALAPPDATA`**

The app stores settings in `~/.claude-terminal/settings.json` (`$HOME/.claude-terminal/`). However, NSIS macros work with `$LOCALAPPDATA`, `$APPDATA`, and `$PROFILE` (which maps to `%USERPROFILE%`).

Best approach for the `customInit`/`customInstall` macro:

```nsis
!macro customInstall
  ${if} ${Silent}
  ${andIfNot} ${isUpdated}
    ; First-time silent install (not an update) - write marker
    CreateDirectory "$PROFILE\.claude-terminal"
    FileOpen $0 "$PROFILE\.claude-terminal\.silent-install" w
    FileWrite $0 "1"
    FileClose $0
  ${endif}
!macroend
```

**Why `customInstall` not `customInit`:** `customInit` runs before files are installed. `customInstall` runs after `installApplicationFiles` and after shortcuts are created (line 81-83 of `installSection.nsh`), making it the right place to write the marker.

**Why `$PROFILE` not `$APPDATA`:** The app uses `os.homedir()` (`%USERPROFILE%`) as the base for `~/.claude-terminal/`. NSIS `$PROFILE` maps to `%USERPROFILE%`.

**Why marker file not registry:** Simpler, cross-platform pattern (even though this is Windows-only), and the app already reads from `~/.claude-terminal/`. No need to add registry reading to the Node.js app.

**Why not command-line arg:** Silent install does NOT launch the app (`isForceRun` is not set), so there is no process to receive the arg. The signal must persist on disk.

### App-Side Detection (HIGH confidence)

In `SetupWizardWindow.js`, modify `isFirstLaunch()`:

```javascript
function isFirstLaunch() {
  try {
    // Check for silent install marker
    const silentMarker = path.join(dataDir, '.silent-install');
    if (fs.existsSync(silentMarker)) {
      // Apply silent install defaults and remove marker
      applysilentInstallDefaults();
      fs.unlinkSync(silentMarker);
      return false; // Skip wizard
    }

    if (fs.existsSync(settingsFile)) {
      const settings = JSON.parse(fs.readFileSync(settingsFile, 'utf8'));
      return !settings.setupCompleted;
    }
  } catch (e) {
    // If we can't read settings, treat as first launch
  }
  return true;
}

function applySilentInstallDefaults() {
  ensureDataDir();
  const defaults = {
    setupCompleted: true,
    hooksEnabled: true,
    launchAtStartup: false
  };

  let existing = {};
  try {
    if (fs.existsSync(settingsFile)) {
      existing = JSON.parse(fs.readFileSync(settingsFile, 'utf8'));
    }
  } catch (e) { /* ignore */ }

  fs.writeFileSync(settingsFile, JSON.stringify({ ...existing, ...defaults }, null, 2));

  // Install hooks (same as wizard "hooks on" path)
  try {
    const HooksService = require('../services/HooksService');
    HooksService.installHooks();
  } catch (e) {
    console.error('Failed to install hooks after silent install:', e);
  }
}
```

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Silent mode detection in NSIS | Custom command-line parsing | `${Silent}` built-in | NSIS native, set automatically by `/S` flag |
| Update vs fresh install detection | Custom registry/file checks | `${isUpdated}` flag | electron-builder already parses `--updated` |
| Post-install app launch control | Custom launch logic | Built-in assisted installer logic | `installSection.nsh` already handles it correctly |
| Page skipping in silent mode | Manual page abort calls | NSIS silent mode behavior | `/S` flag automatically skips all wizard pages |

## Common Pitfalls

### Pitfall 1: Using `customInit` for File Operations
**What goes wrong:** `customInit` runs during `.onInit`, before any files are installed or directories guaranteed to exist.
**Why it happens:** Developers assume init = after install.
**How to avoid:** Use `customInstall` macro for any file writes. It runs after `installApplicationFiles` in `installSection.nsh`.
**Warning signs:** Marker file not appearing after install.

### Pitfall 2: `$APPDATA` vs `$PROFILE` vs `$LOCALAPPDATA`
**What goes wrong:** Writing to `$APPDATA` creates the file in `%APPDATA%` (Roaming), but the app uses `os.homedir()` which is `%USERPROFILE%`.
**Why it happens:** NSIS has multiple "home" variables with different meanings.
**How to avoid:** Use `$PROFILE` in NSIS, which maps to `%USERPROFILE%` = `os.homedir()` in Node.js.
**Warning signs:** Marker file exists but app doesn't find it.

### Pitfall 3: Forgetting the `${isUpdated}` Guard
**What goes wrong:** Marker file is written on every silent install, including updates. App re-applies defaults on every update.
**Why it happens:** Updates also run silently with `/S` (from `autoInstallOnAppQuit`).
**How to avoid:** Always check `${andIfNot} ${isUpdated}` alongside `${Silent}`.
**Warning signs:** User settings reset after auto-update.

### Pitfall 4: Async Hook Installation
**What goes wrong:** `HooksService.installHooks()` is async but `isFirstLaunch()` is synchronous.
**Why it happens:** The wizard uses async flow, but marker detection runs in sync init code.
**How to avoid:** Either make the hook installation fire-and-forget (catch errors), or restructure to handle the async flow in `initializeApp()` instead of `isFirstLaunch()`.
**Warning signs:** Hooks not installed after silent install, or unhandled promise rejection.

### Pitfall 5: Silent Uninstall Desktop Shortcut
**What goes wrong:** The existing `customUnInstall` macro already handles this correctly by checking `${isUpdated}`.
**Why it happens:** N/A - already correct.
**How to avoid:** Don't change `customUnInstall`. It deletes the desktop shortcut on real uninstall, preserves on update. Silent uninstall is already supported by NSIS `/S` flag.

### App-Side Silent Install Detection

```javascript
// In SetupWizardWindow.js or a new utility

const path = require('path');
const fs = require('fs');
const { dataDir, settingsFile, ensureDataDir } = require('../utils/paths');

const SILENT_MARKER = path.join(dataDir, '.silent-install');

function handleSilentInstall() {
  if (!fs.existsSync(SILENT_MARKER)) return false;

  // Clean up marker immediately
  try { fs.unlinkSync(SILENT_MARKER); } catch (e) { /* ignore */ }

  // Apply defaults
  ensureDataDir();
  const defaults = {
    setupCompleted: true,
    hooksEnabled: true,
    launchAtStartup: false
  };

  let existing = {};
  try {
    if (fs.existsSync(settingsFile)) {
      existing = JSON.parse(fs.readFileSync(settingsFile, 'utf8'));
    }
  } catch (e) { /* ignore */ }

  // Don't overwrite if settings already exist (shouldn't happen, but safety)
  if (!existing.setupCompleted) {
    fs.writeFileSync(settingsFile, JSON.stringify({ ...existing, ...defaults }, null, 2));
  }

  return true;
}
```

## Open Questions

1. **Hook installation timing**
   - What we know: `HooksService.installHooks()` is async. The setup wizard calls it in an async handler.
   - What's unclear: Can we safely call it synchronously or fire-and-forget in the silent install path?
   - Recommendation: Call it from `initializeApp()` (which is the sync entry point in main.js) as a fire-and-forget promise. If it fails, hooks can be installed manually later.

2. **UpdaterService.quitAndInstall() parameters**
   - What we know: Currently calls `autoUpdater.quitAndInstall()` with no args (isSilent=false, isForceRunAfter=false). This means manual "Install Update" shows the wizard briefly.
   - What's unclear: Should we pass `(true, true)` for a seamless update experience?
   - Recommendation: This is out of scope for Phase 30 (which focuses on the `/S` flag fix), but worth noting for a future update UX improvement. The `autoInstallOnAppQuit` path already uses `install(true, false)` which is correct.
