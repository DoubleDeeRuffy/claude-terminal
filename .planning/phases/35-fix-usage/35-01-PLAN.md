  <action>
The bug: The Anthropic OAuth API returns `utilization` as a decimal fraction (0.0 to 1.0), e.g. `five_hour.utilization = 0.41` means 41%. The code in `fetchUsageFromAPI()` (line 87-90) passes these raw decimals through. Then `updateUsageBar()` in renderer.js does `Math.round(percent)` which rounds 0.41 to 0 (or 0.009 to 0, 0.29 to 0, etc.).

Fix in `fetchUsageFromAPI()` (UsageService.js lines 85-95): Convert API decimal utilization values to percentages by multiplying by 100. Apply this to `session`, `weekly`, `sonnet`, and `opus` fields:

```javascript
session: json.five_hour?.utilization != null ? json.five_hour.utilization * 100 : null,
weekly: json.seven_day?.utilization != null ? json.seven_day.utilization * 100 : null,
sonnet: json.seven_day_sonnet?.utilization != null ? json.seven_day_sonnet.utilization * 100 : null,
opus: json.seven_day_opus?.utilization != null ? json.seven_day_opus.utilization * 100 : null,
```

Use `!= null` (not `?? null`) to ensure we only multiply when a real number is present. This preserves null for missing data.

Do NOT change the PTY path — `parseUsageOutput()` already extracts integer percentages from the `/usage` command text output (regex captures digits before `%`).

Do NOT change `updateUsageBar()` in renderer.js — `Math.round(percent)` is correct once it receives values in the 0-100 range.

Add a debug log right after the conversion to confirm values during development:
```javascript
console.log('[Usage] Converted utilization:', { session: ..., weekly: ..., sonnet: ..., opus: ... });
```
