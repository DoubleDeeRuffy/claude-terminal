```javascript
function writePreservingScroll(terminal, data) {
  const buf = terminal.buffer.active;
  const wasScrolledUp = buf.viewportY < buf.baseY;
  const savedOffset = buf.baseY - buf.viewportY;
  terminal.write(data);
  if (wasScrolledUp) {
    const newTarget = terminal.buffer.active.baseY - savedOffset;
    const delta = newTarget - terminal.buffer.active.viewportY;
    if (delta !== 0) terminal.scrollLines(delta);
  }
}
```

    **New implementation strategy:**

    1. Add module-level state for tracking scroll restoration per terminal:
    ```javascript
    // Map of terminal -> { scrollRestoreTimer, savedOffset, wasScrolledUp }
    const scrollRestoreState = new WeakMap();
    ```

    2. Replace `writePreservingScroll(terminal, data)` with this logic:
       - Read `buf.viewportY < buf.baseY` to detect if user has scrolled up
       - If scrolled up, capture `buf.baseY - buf.viewportY` as the saved offset
       - Call `terminal.write(data)` with NO immediate scroll restoration (no `scrollLines()` call)
       - After write, if user was scrolled up, set/reset a debounce timer (~80ms). When the timer fires (output has settled), read the NEW `buffer.active.baseY`, compute `newTarget = baseY - savedOffset`, compute delta from current `viewportY`, and call `terminal.scrollLines(delta)` if non-zero
       - If user was NOT scrolled up, do nothing (let xterm auto-scroll to bottom as normal)
       - Clear any pending timer if new data arrives before it fires (this is the key — rapid output keeps resetting the timer, so scroll restoration only happens once output pauses)

    4. **Edge case: buffer type switch.** If `terminal.buffer.active.type` changes between when offset was saved and when the timer fires (Claude entering/leaving alternate screen), discard the saved offset and do NOT call `scrollLines()`. Check `buffer.active.type` at save time and at restore time; if they differ, skip restoration.

    5. **Edge case: basic terminals with slow output.** The debounce approach works correctly for slow output too — if data arrives slowly (>80ms gaps), each write will trigger scroll restoration after 80ms, which is the same UX as the current immediate approach but without the per-write flicker during rapid output. No special handling needed.

    6. Use 80ms as the debounce delay. This is above the main-process flooding batch interval (50ms) so rapid output will consistently reset the timer, but short enough to feel responsive when output pauses.
